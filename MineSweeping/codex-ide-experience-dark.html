01 / 初始体验

主要痛点不是单点功能缺失，而是体验重心错位

Qoder、Cursor、VS Code Copilot 更像贴在 IDE 里的副驾驶；Codex 更像项目级 agent，擅长读仓库、改文件、跑命令、接 MCP、用 skills。

权限太保守如果启动在 read-only，Codex 会变成只能看不能顺手做的顾问。
审批太频繁读文件、联网、PowerShell、MCP 反复确认，会带来明显疲劳感。
项目记忆为空没有项目背景和长期约定时，协作容易像临时外包。
风格未明确AGENTS.md 和项目说明决定它是“接单”还是 pair programmer。
02 / 配置基线

Codex 端适合日常开发的安全默认

推荐配置让 Codex 可以在工作区内完成普通修改，真正越界或失败时再请求确认。

sandbox_mode = "workspace-write"
approval_policy = "on-failure"
web_search = "live"
  • Codex 可以在当前项目工作区内读写文件。
  • 普通项目内修改不应反复审批。
  • 命令失败后需要更高权限时再询问。
  • 联网搜索启用 live web search。
  • 工作区外写入、依赖安装、破坏性操作、凭据修改仍然要审批。

只读审查模式

sandbox_mode = "read-only"
approval_policy = "never"
web_search = "live"
03 / IDE 身体

Qoder、Cursor、Copilot 的安心感来自 IDE 深度集成

这些工具能直接使用当前文件、光标、选区、打开的 tab、diagnostics、文件树、Git 状态、diff editor、terminal、preview / webview 和 workspace index。

即时编辑ghost text 补全、inline edit、当前选区变换。
可视化审阅工作区红绿 diff、gutter change marker、文件树 dirty 状态。
逐块决策部分工具支持 accept / reject 和 AI 预览 diff。
上下文注入项目索引、repo memory、当前编辑器上下文自动进入对话。

这些能力不是单靠模型获得的，而是 IDE extension 与编辑器 API 深度绑定后的结果。

04 / Agent 大脑

Codex 的优势是项目级 agent 能力

  • 跨文件理解和修改。
  • 用 shell 跑检查和测试。
  • 通过 MCP 连接外部系统。
  • 通过 skills 固化工作流。
  • 使用更强模型做推理、规划、重构和审查。
  • 适合较长任务和项目级任务。

但 Codex 桌面 / CLI 本身不是 IDE，所以天然不等价于原生 ghost text、编辑器内光标级补全、主编辑区 inline accept / reject。

05 / Diff 三层

这轮测试里最容易混淆的是 diff

1

Git 工作区 diff

Codex 修改磁盘文件，IDE 文件监听发现变化，Git 发现工作区相对 HEAD 变化，然后文件树、gutter、Source Control 显示红绿标记。

Codex 修改磁盘文件
-> IDE 文件监听发现变化
-> Git 发现工作区相对 HEAD 变化
-> 文件树、gutter、Source Control 显示红绿标记
2

Git diff editor

在 Source Control / Git 面板里点开文件看到的只读红绿对比，适合审阅、复制、回退、比较。

3

AI 预览 diff

AI 在真正写入文件之前展示建议改动，让用户逐块 accept / reject。它需要插件维护 pending edits 状态。

06 / Qoder 侧栏

Qoder 里的 Codex 侧栏 diff 更接近 agent 审批流

从 Qoder 内部 Codex 侧栏发起修改时，侧栏会弹出自己的小窗显示 diff。它能展示修改，但选择在侧栏 UI 管理，而不是完全放到主工作区 inline diff。

侧栏任务
-> agent 生成修改
-> 插件展示 diff
-> 用户审批 / 应用
-> 文件落盘
07 / MCP 定位

MCP 是工具和数据协议，不是 IDE UI 协议

MCP 适合补GitHub issue / PR、数据库、浏览器测试、文档和私有知识库、日历、Notion、Google Drive、内部 API、搜索和资料查询。
MCP 不适合直接补ghost text、主编辑器 inline diff、gutter UI、文件树 dirty marker、编辑器里的 accept / reject 控件。
IDE 负责 UI 和即时反馈
Codex 负责 agent 能力和外部工具
MCP 负责连接外部世界
AGENTS.md / skills 负责工作风格和流程记忆
08 / 推荐使用

最佳路线不是让 Codex 替代 Qoder / Cursor，而是组合它们

日常开发

  1. 用 Qoder / Cursor / VS Code 打开项目。
  2. 同时让 Codex 打开同一个目录。
  3. 让 Codex 负责跨文件分析、修改、测试。
  4. 在 IDE 主工作区看 dirty 状态、gutter、Git diff。
  5. 用 IDE 自己的撤销、回退、diff UI 审阅。

小范围当前文件编辑

改几行、补函数、调整命名、当前选区变换、想逐块 accept / reject 时，适合用 IDE 原生 AI inline edit。

大范围项目任务

梳理架构、跨文件重构、增加功能、调试复杂错误、跑测试并修复，更适合交给 Codex。

09 / 项目记忆

建议补齐项目记忆层

AGENTS.md
.codex/project-memory.md
.codex/repo-map.md
.codex/tasks.md
  • AGENTS.md: 项目规则、命令、风格、禁忌。
  • .codex/project-memory.md: 长期记忆，架构决定、坑、约定。
  • .codex/repo-map.md: 项目地图，入口、模块、数据流。
  • .codex/tasks.md: 当前任务、已知问题、下一步。
项目初始化 skill自动扫文件树、识别语言框架、找启动和测试命令、生成 repo map。
固定 diff 总结改完后说明文件、行为、验证和风险。
Browser 验收前端项目启动 dev server，用浏览器打开 localhost 并检查布局。
MCP 精简安装优先装真正会用的连接，减少噪音和审批。
10 / 最终结论

最现实、最舒服的目标是组合，而不是替代

Qoder / Cursor / VS Code 负责 IDE 身体
Codex 负责高能力 agent 大脑
MCP 负责连接外部工具和数据
AGENTS.md / skills 负责个性、记忆和流程
Git / IDE diff 负责可视化审阅
  • 有 IDE 的安心感。
  • 有 Codex 的模型和 agent 能力。
  • 有 MCP 和 skills 的生态。
  • 有项目级记忆。
  • 有工作区 diff 可审阅。
  • 减少无意义审批。
  • 保留关键安全边界。
不要把 Codex 当成 IDE 的替代品。把它配置成 IDE 旁边的高能力协作脑。
